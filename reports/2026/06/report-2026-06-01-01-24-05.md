##### [#6222](https://github.com/microsoft/agent-framework/pull/6222) — .NET: Fix Magentic to share agent replies across team

- 作者: ghominejad / 状態: OPEN / マージ日時 (UTC): `—`
- ラベル: `.NET`, `workflows`
- 規模: `+169/-7`

**変更概要**

`MagenticOrchestrator` が **per-round の指示を untargeted に送る → fan-out edge が全員配布、参加者の返答は誰にもリレーされない** という設計バグの修正。同じアセンブリの `GroupChatHost` と Python 版 Magentic の挙動とまったく逆になっており、コラボ系のハンドオフが破綻していた。Issue [#6223](https://github.com/microsoft/agent-framework/issues/6223) を fix する形で同時着地。

**コミットレベルの詳細**

- per-round の指示を **選定された speaker 一人にターゲット送信** に変更。
- 各参加者の reply を、応答者本人を除いた他参加者に **context として broadcast** (TurnToken なしのバッファ送信、`_currentSpeakerExecutorId` で responder を除外)。GroupChatHost のパターンに一致。
- `_currentSpeakerExecutorId` を **checkpoint 越しに永続化** し、リストア後も除外が機能するように。
- `RecordingEchoAgent` を使った回帰テストを追加。「2 番手 speaker が 1 番手の `instruction` ではなく `response` を見る」ことを assert。

**関連 Issue / Discussion**

- Issue [#6223](https://github.com/microsoft/agent-framework/issues/6223) (本 PR が fix)
- 関連: 継続ウォッチ中の [#6173](https://github.com/microsoft/agent-framework/issues/6173) (Magentic が常時 MaxRounds に達する) — 本修正でコンテキスト伝播が直れば連動して解消する可能性

**既存利用者への影響**

`MagenticOrchestrator` ベースのワークフローを組んでいる .NET ユーザは、マージ後は各参加者の `ChatHistory` に流入するメッセージ集合が **修正前と非互換** で変化する (他人の "指示" は来なくなる、他人の "応答" が来るようになる)。マネージャの ChatHistory には影響なし。

#### 主要な変更点

- **Kestrel の accept-loop が ILogger 例外で永久ハングする回帰を新規発覚 + 修正 PR 着手**。Issue [#66942](https://github.com/dotnet/aspnetcore/issues/66942) は `ConnectionDispatcher<T>` の `while(true)` 全体を 1 つの try/catch で囲んでいるためロギング層が例外を投げただけで accept ループが永久に抜けてしまい、プロセスは生存しつつ新規接続を受け付けなくなる silent hang を指摘。修正 PR [#66943](https://github.com/dotnet/aspnetcore/pull/66943) は AcceptAsync 失敗時の critical ログ + 既存セマンティクスを維持しつつ、accept 後の setup / 接続作成 / 処理を per-iteration の try/catch に切り出し、失敗した接続は abort/dispose して accept を継続する形に再構成。
- **OpenAPI スキーマ ID 衝突修正** ([#66937](https://github.com/dotnet/aspnetcore/pull/66937), fix [#64325](https://github.com/dotnet/aspnetcore/issues/64325))。異なる namespace の同名型が同一 schema ID に解決されてしまうケースで参照 ID を重複排除。`CreateSchemaReferenceIdTests` の非 nullable DateTime テストも併せて修正。先行 PR [#65871](https://github.com/dotnet/aspnetcore/pull/65871) は本 PR に置き換えられて close。

- **Aspire CLI を npm パッケージとして配布開始 ([#17297](https://github.com/microsoft/aspire/pull/17297) merged)**。`+5013/-320` 行、43 ファイル変更。`@microsoft/aspire-cli` を「ポインタ」パッケージとし、`@microsoft/aspire-cli-linux-x64` / `-win-x64` / `-osx-arm64` などの RID 固有パッケージから実バイナリを引く構造。tarball は既存の署名済み native CLI アーカイブから生成し、**byte-for-byte の検証を CI で実施してから ESRP/MicroBuild の Azure DevOps release pipeline で公開** (RID 群を先に publish、レジストリ伝播待機後にポインタを publish)。`aspire update` も npm 経由インストールを検知し `npm update -g @microsoft/aspire-cli` を案内。`docs/specs/npm-cli-package.md` と `docs/release-process.md` にレイアウト・バージョニング・recovery flags・セキュリティ考慮事項を整備。後続の改善要望が複数 ([#17767](https://github.com/microsoft/aspire/issues/17767) exit code / supportedRids drift / cache layout)。
- **darc feed ルーティング修正 ([#17743](https://github.com/microsoft/aspire/pull/17743) merged)** で staging-identity CLI を darc feed に向ける。これに連なる `release/13.4` への backport である **[#17766](https://github.com/microsoft/aspire/pull/17766)** と npm signing scope 修正 [#17770](https://github.com/microsoft/aspire/pull/17770) で 13.4 リリース直前の調整中。

- **「Agent リソース」抽象を新規提案 ([#17757](https://github.com/microsoft/aspire/pull/17757))** — `builder.AddPythonApp(...).AsAgent(AgentProtocol.A2AJsonRpc)` のように `AsAgent(AgentProtocol)` でエージェント扱いを宣言可能に。A2A、OpenAI Responses、AG-UI、ACP の各プロトコルを統一モデルで扱い、Dashboard でプロトコル別 URL と send-message コマンドをハイライト表示。MCP サーバ要素は今までどおり `WithMcpServer()` 経由で分離。Issue [#17752](https://github.com/microsoft/aspire/issues/17752) が同時提案者から fix リンク付きで提出されている。

- **内部 CI 高速化 ([#17760](https://github.com/microsoft/aspire/pull/17760))** — `microsoft-aspire` パイプラインの wall-clock を 121 min → 約 57 min (≈ 53% 削減) に短縮。Split C+A、narrow restore + parallel downloads、Linux Assemble、SkipPlayground の段階的改善。出荷物は一切変えず、ビルドグラフとエージェント選定のみの最適化。
